# 架构决策记录 (ADR)

## ADR-001: 采用 Clean Architecture

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要选择一个可维护、可测试、可扩展的架构模式。

**决策**:
采用 Clean Architecture 三层架构：
- Presentation 层：UI 和 ViewModel
- Domain 层：业务逻辑和 UseCase
- Data 层：数据源和 Repository

**理由**:
1. 关注点分离
2. 易于测试
3. 独立于框架
4. 易于维护和扩展

**后果**:
- 需要更多的样板代码
- 学习曲线较陡
- 但长期收益大

## ADR-002: 使用 Jetpack Compose

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要选择 UI 框架。

**决策**:
使用 Jetpack Compose 而不是传统的 XML 布局。

**理由**:
1. 声明式 UI
2. 更少的代码
3. 更好的性能
4. Google 官方推荐
5. 现代化开发体验

**后果**:
- 需要学习新的 UI 范式
- 最低支持 API 21
- 但开发效率更高

---

## ADR-003: 手动依赖注入

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要选择依赖注入方案。

**决策**:
使用手动依赖注入，不使用 Hilt 或 Dagger。

**理由**:
1. 项目规模较小
2. 减少依赖
3. 更容易理解
4. 更快的编译速度

**后果**:
- 需要手动管理依赖
- 大型项目可能不适用
- 但对当前项目足够

## ADR-005: 使用 SharedPreferences

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要选择本地存储方案。

**决策**:
使用 SharedPreferences 存储简单数据。

**理由**:
1. 简单易用
2. 适合存储少量数据
3. 系统内置
4. 无需额外依赖

**后果**:
- 不适合大量数据
- 不支持复杂查询
- 但满足当前需求

## ADR-006: 不使用数据库

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要决定是否使用数据库。

**决策**:
当前不使用 Room 或其他数据库。

**理由**:
1. 当前只需存储少量数据
2. SharedPreferences 足够
3. 减少复杂度
4. 可以后续添加

**后果**:
- 无法存储复杂数据
- 无法进行复杂查询
- 但保持简单

## ADR-007: 使用 StateFlow

**状态**: 已接受

## ADR-010: 中英文双语支持

**日期**: 2024-01-01

**状态**: 已接受

**背景**:
需要决定是否支持多语言。

**决策**:
支持中英文双语。

**理由**:
1. 扩大用户群
2. 展示国际化能力
3. 易于添加更多语言

**后果**:
- 需要维护多份资源文件
- 但提升用户体验
